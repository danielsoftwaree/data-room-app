# SPEC.md — Продуктовая спецификация: Data Room MVP

Этот документ — источник ответов на вопросы вида «что входит в MVP?», «зачем этот продукт?»,
«что мы сознательно не делаем?». Технические решения и порядок работ — в `PLAN.md`,
архитектура — в `docs/architecture.md`.

## 1. Позиционирование

**Data Room** — виртуальная комната данных: организованное хранилище документов для due diligence
(вдохновение: Google Drive / Dropbox / Box, где Data Room — верхнеуровневый «диск»).

- **Для кого:** команда сделки (Acme Corp.), которой нужно собрать и структурировать документы
  для многомиллиардной сделки поглощения.
- **Какую задачу решает:** быстро разложить документы по понятной иерархии папок, находить их
  и просматривать — без обучения и без инструкции.
- **Что является продуктом на самом деле:** это тестовое задание; продукт одновременно демонстрирует
  UX-мышление, инженерную зрелость (модель данных, edge cases) и AI-centric workflow разработки.
  Data Room — прокси-домен для CRM/ERP-реальности (иерархии, CRUD, списки).

## 2. Принципы продукта

Приоритеты — в порядке, заданном заказчиком тестового:

1. **UX и функциональность прежде всего.** Интуитивные флоу без макета и дизайнера: продуманные
   состояния (пустая папка, загрузка, ошибка), навигация без тупиков, подтверждение деструктивных
   действий.
2. **Дизайн и полировка — вторыми.** Чисто, аккуратно, консистентно. **Никаких мёртвых кнопок:**
   нереализованная фича не должна присутствовать в UI.
3. **Качество кода — третьим.** Чистая доменная модель, но код не важнее продукта.
4. **Zero-config опыт.** MVP «works well out of the box»: клонировал → запустил → работает.
5. **Честность в трейд-оффах.** Все сознательные упрощения фиксируются в README
   как design decisions, а не прячутся.
6. **Маленький, но законченный.** Лучше меньше фич, но каждая доведена до конца.

## 3. Scope

### В объёме продукта (текущий горизонт)

- Управление датарумами (верхний уровень «диска»).
- Иерархия папок произвольной вложенности внутри датарума.
- Файлы: только **PDF**.
- Просмотр PDF прямо в UI.
- Навигация: breadcrumbs, вход в папки, URL отражает текущее положение.
- Постоянное (durable) хранилище: метаданные и PDF-блобы в **PostgreSQL** (Drizzle ORM);
  данные переживают и refresh вкладки, и рестарт сервера.
- Обработка edge cases: дубликаты имён, невалидные/пустые/длинные имена, не-PDF при загрузке,
  каскадное удаление с предупреждением и подсчётом содержимого, частичные ошибки при
  множественной загрузке.

### Вне объёма (см. также Non-goals)

- Любые типы файлов кроме PDF.
- Совместная работа: шаринг, права доступа, роли, комментарии.
- Управляемое облачное blob-хранилище (S3 и т.п.): PDF-блобы храним в той же PostgreSQL
  (bytea) — достаточно для масштаба тестового.

## 4. MVP — что входит

MVP = обязательный функционал тестового задания, доведённый до законченного состояния.

### 4.1. Датарумы
- Создать датарум.
- Список датарумов, открытие датарума.

### 4.2. Папки
- Создать папку; вкладывать папки в папки (произвольная вложенность).
- Просматривать папку и её содержимое (вложенные папки и файлы).
- Переименовать папку.
- Удалить папку с каскадом (все вложенные папки и файлы) — с подтверждением
  «будет удалено N папок и M файлов».

### 4.3. Файлы
- Загрузка только PDF (drag&drop + кнопка), валидация по MIME и расширению.
- Просмотр PDF в UI.
- Переименование файла.
- Удаление файла.

### 4.4. Навигация и состояния
- Breadcrumbs и вход в папки; URL-состояние (обновление страницы не теряет позицию).
- Пустые состояния с CTA, индикаторы загрузки, внятные сообщения об ошибках.

### 4.5. Edge cases (обязательная часть MVP)
- Дубликат имени файла/папки в одной папке (включая переименование в занятое имя;
  авто-суффикс «file (1).pdf», case-insensitive).
- Пустое/пробельное имя, ограничение длины, запрещённые символы.
- Не-PDF файл при загрузке — внятная ошибка.
- Удаление непустой папки — предупреждение с подсчётом.
- Множественная загрузка с частичными ошибками.

### 4.6. Полировка (в рамках тайм-бокса MVP)
- Тосты на операции, скелетоны/спиннеры.
- Контекстное меню на строке (rename/delete).
- Сортировка: папки сверху, затем по имени.
- Базовая доступность (aria от shadcn/Radix).

**Критерий готовности MVP:** решение работает end-to-end, все пункты 4.1–4.5 закрыты,
в UI нет нереализованных элементов, README описывает design decisions и setup.

## 5. Фазы продукта

**Фаза 0 — Фундамент (сделано).**
Монорепозиторий (Bun + Turborepo), apps/web + apps/api, доменная модель и бизнес-правила
в `@repo/domain`, реальные CRUD-эндпоинты (изначально in-memory store), генерируемый
типизированный клиент (OpenAPI → Orval → TanStack Query).

**Фаза 1 — MVP (текущая).**
PostgreSQL + Drizzle ORM (замена in-memory store; см. PLAN.md, раздел «База данных»),
затем раздел 4 целиком: папки + навигация → файлы → edge cases + полировка → README.

**Фаза 2 — Публичность (extra credit, только при остатке времени).**
- Деплой frontend + backend (Vercel) — рекомендован заданием, делаем в первую очередь из фазы 2.
- Поиск по имени в пределах датарума.

**Фаза 3 — Гипотетическое развитие (вне тайм-бокса; направление, не обязательство).**
- Вынос PDF-блобов из Postgres в облачный blob storage (S3/R2).
- Аутентификация (social auth или логин/пароль).
- Поиск по содержимому документов, фильтры.

## 6. Non-goals

Сознательно **не делаем** (и честно отмечаем в README как out of scope):

- **Аутентификация и авторизация** — extra credit; в тайм-бокс не входит.
- **Облачный blob storage (S3 и т.п.)** — PDF-блобы живут в PostgreSQL (bytea);
  вынос в объектное хранилище — фаза 3.
- **Поддержка не-PDF форматов** — задание явно ограничивает загрузку PDF.
- **Шаринг, права, роли, аудит, версионирование документов** — атрибуты «взрослых» data room,
  не MVP.
- **Поиск по содержимому** — extra credit фазы 3; в MVP нет и поиска по имени.
- **Реалтайм/мультиюзер** — один пользователь, одна вкладка как основной сценарий.
- **Мобильное приложение** — только адаптивный web в разумных пределах.
- **Фичи «на вырост» в UI** — никаких кнопок-заглушек и меню без реализации (принцип №2).
